The Problem

AI coding tools are fast.
That speed is also the risk.

When your AI can write 500 lines in a minute, it's easy to start building before you've answered the hard questions. Three hours later you're refactoring in code review — or worse, starting over.

The team already knows the right process. SpecGantry makes it impossible to skip it.

Speed without direction

AI writes code instantly, but it doesn't know if you're solving the right problem until after the spec review.

Architecture discovered too late

Violations found in code review cost 10× more to fix than violations caught at spec time.

Invisible token spend

AI loads hundreds of lines of context per feature with no visibility into what each capability actually cost.

Architecture drift across capabilities

Each capability re-describes entities, permissions, and contracts without challenge. Three capabilities later, the data model has three different definitions of the same entity.

The Solution

One loop. Three phases. Quality challenged before it moves on.

v7 introduces an adversarial Challenge-Write-Judge loop at every phase. A challenger asks what would block the next phase — specific to this project, not a generic checklist. The judge asks one question: "would the next phase be blocked?" Gaps caught at ideation cost nothing to fix. Gaps caught at spec save a full build cycle. Gaps caught at code are repaired before shipping.

01
Ideation CWJ Loop
challenge → user answers all questions per round → judge readiness → write all artifacts on CLEAR · north-star.md · architecture.md · intent.md per capability
02
Spec CWJ Loop — per capability
developer-proxy challenge → write capability-spec.md → judge → user approves once on CLEAR
03
Code CWJ Loop — per capability
plan build approach → build end-to-end → user-proxy challenge · gap classified as code or spec → repair route · build-report.yaml
04
Deploy Release
north-star alignment check · changelog updated · Dockerfiles · deploy.sh · release versioned
05
[N] New Work — post-release
Investigate → classify CODE_BUG / SPEC_GAP / REQUIREMENT_DRIFT / NEW_CAPABILITY → route to the right phase
Three challenger identities: ideation ("what would stop a developer agreeing to start?") · spec ("what would a developer be blocked on building from this?") · code ("can a user actually accomplish what was promised?"). Each reads the per-project north star and challenges against it.
Specs are the memory: all knowledge lives in specs/. North star, architecture decisions, capability intent, what was built — all in plain-text YAML and Markdown, committed to git. Agents read from canonical artifacts on disk. Nothing is duplicated in transition parameters.
Key Features

Everything you need to ship AI-assisted features with confidence

Engagement Hooks — Claude Stays On-Process

Project setup writes a SessionStart hook into .claude/settings.json. Claude Code injects the engagement contract automatically on every session start — so Claude never forgets to route through /spec-gantry, even deep into a long conversation.

Single Architecture File — Written Once, Referenced Everywhere

All technical decisions live in one specs/architecture/architecture.md with ## section:name anchors. Data model, actors, API interfaces, deployment, guardrails, and configuration — all in one file. Every capability spec reads only the sections it needs. Architecture drift is structurally impossible.

How the architecture file works →

Challenge Before Code — Hard Gates

No code is written until a capability spec exists that was challenged by a developer-proxy and approved by a judge. The gate reads the filesystem — not a flag, an artifact.

Per-Project North Star — Not a Template

Each project gets one specs/north-star.md written from the actual idea during ideation. Flowing prose, no headings. Every challenger and judge reads it whole. It grows across the lifecycle as new requirements surface.

Diagnostic Routing — Fix the Right Thing

When something is wrong, the investigate agent classifies the root cause as CODE_BUG / SPEC_GAP / REQUIREMENT_DRIFT / NEW_CAPABILITY before routing. A spec gap sent to the code loop produces another iteration of the wrong repair. Classification ensures the right phase gets fixed.

intent.md — Why Every Capability Exists

Every capability gets a 2-paragraph intent.md — the experience promise. The code challenge agent reads it to trace the user's experience. Every source file gets a @capability and @intent anchor. Investigation never needs to load full specs to understand purpose.

Reverse-Engineer Existing Codebases

SpecGantry analyses your existing code and generates north-star.md, architecture.md, a capability list, intent files, stub specs, stub build reports, and anchor tags. Bring structure to any existing project in one command.

See reverse engineering →

Developer Intelligence — Not Just Cost

Every agent run logs tokens and cost. The insights view adds iteration counts per capability per phase, challenge density, outlier detection, and release comparison. Know which capabilities were expensive, how many cycles each phase took, and what kept getting challenged.

See developer intelligence →

Changelog for Release Safety

An append-only specs/changelog.md tracks dropped and deprecated fields across releases. Spec write and code plan agents read it before referencing any field or interface — preventing new release capabilities from using removed APIs.

Build Report — What Was Built and How to Test It

The build agent writes a build-report.yaml per capability — overall status, quality block with iteration count and exit reason, runtime info (source_root, language, exposed ports), and a test plan with one observable verification per criterion.

Crash-Safe Resumption

Every phase transition is written to disk immediately. A crash mid-session loses at most one in-progress answer — everything confirmed before that is on disk and the pipeline resumes exactly where it left off.

2 Skills · 12 Agents

One command. Three adversarial loops. Every workflow covered.

/spec-gantry
Your Single Entry Point — New project, existing codebase, daily work, bug fixes, new capabilities. Run this at the start of every session. SpecGantry reads your state, shows the full pipeline dashboard, and routes you to the right agent automatically.
/track-cost
Developer Intelligence — Cost by Challenge/Write/Judge columns, by capability, by release. Iteration counts, challenge density, outlier detection. Know exactly what each capability cost and how many cycles it took.

The 12 Agents — 3 per CWJ phase

ideation-challenge
Haiku
Adversarial challenger. Reads vision and prior answers, then fires up to 7 blocking questions specific to this project — not a fixed script. A recipe manager gets different questions than a multi-tenant SaaS. Never writes files.
ideation-judge
Haiku
Unblock checker. Asks one question: "Could a developer start writing capability specs without inventing answers?" Six criteria covering capability list, data model, tech decisions, UX intent, and answer completeness. Returns CLEAR or BLOCKED with specific gaps.
ideation-write
Sonnet
Consolidation agent — runs once on CLEAR, not per cycle. Writes north-star.md (per-project flowing prose), architecture.md (all technical decisions, ## section:name anchors), and intent.md per capability. Crash-safe write order with recovery guards.
spec-challenge
Haiku
Developer-proxy challenger. Reads north star and capability intent, then asks: "If I were a developer about to build this, what questions would I have?" Covers loading states, empty states, error states, output format, navigation flow, and north-star alignment.
spec-write
Sonnet
Writes capability-spec.md resolving every challenge into observable, bounded, unambiguous criteria. May extend north-star.md if a new requirement surfaces. On iteration 2+ amends rather than rewrites.
spec-judge
Haiku
Unblock checker. Traces the user flow through the written spec. Asks whether a developer reading it would hit a wall. Checks triggers, async states, output format, error messages, flow completeness, and north-star alignment. Emits the approval summary the user signs off on.
code-plan
Sonnet
On iteration 1: builds the implementation approach — layer order, async patterns to apply upfront, what to reuse, the riskiest part and how to de-risk it. On iteration 2+: classifies the challenge agent's findings as code/spec/design failures and plans targeted repair.
code-build
Sonnet
Builds the capability end-to-end from the plan. Writes @capability/@intent/@entry/@contract anchors. Applies contract pre-flight against architecture.md api-interfaces section. Writes build-report.yaml with source_root, language, and test_plan.
code-challenge
Haiku
User-proxy challenger. Reads north star, intent, and source files (NOT the spec). Traces the user's experience through the actual code. Classifies each gap as "code" (spec implied it, code missed it) or "spec" (north star requires it, spec never captured it). Classification drives the repair route.
investigate
Haiku
Diagnostic classifier. For any post-deploy problem, classifies the root cause as CODE_BUG / SPEC_GAP / REQUIREMENT_DRIFT / NEW_CAPABILITY before routing. Locates files as supporting evidence. Each classification routes to a different repair phase.
deployment
Sonnet
North-star alignment check first — verifies what's shipping delivers what was promised, surfaces non-blocking warnings for capped capabilities. Then generates per-service Dockerfiles, docker-compose.yml, .env.example, and a versioned deploy.sh. Supports GCP Cloud Run, AWS ECS/Fargate, Azure Container Apps, and Docker Compose.
reverse-engineer
Haiku
Synthesises a full SpecGantry v7 structure from an existing codebase — north-star.md, architecture.md, capability list, intent.md per capability, stub capability-specs, stub build reports, and @capability/@intent/@entry/@contract anchor tags in source files.
