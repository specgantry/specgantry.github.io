The Problem

Speed-first AI development has a predictable failure mode.

You describe what you want. The AI builds something fast. Then someone asks why the error state isn't handled, why the data model contradicts itself, or why the build doesn't match what you agreed on. The code is fine. The problem is everything that wasn't decided before it ran.

Ambiguity baked in at the start

Missing loading states, undefined error messages, unresolved scope edges. These aren't bugs the AI introduces — they're decisions nobody made before the build started.

Architecture drift across capabilities

Without one authoritative architecture file, each capability invents its own interpretation. Three capabilities in, you have three conflicting definitions of the same entity.

Rework routed to the wrong phase

A spec gap sent to the code repair loop will never converge — the spec must be fixed first. Without root cause classification, you keep re-running the wrong loop.

Invisible AI spend

Token spend per capability, per phase, per repair cycle — invisible unless you build the instrumentation yourself. You can't manage what you can't see.

How It Works

Four phases. An adversary at every gate.

Before each phase produces anything, an independent agent challenges it — asking what would block the next phase if work started now. Gaps caught at ideation cost nothing to fix. Gaps caught in production cost the most.

01
Ideation
You answer · high interaction

Claude challenges your idea before anything is written. You answer; it commits a north star and architecture every later phase reads.

02
Spec
Autonomous · you approve once

A developer-proxy finds what would block building this. You approve once — after a judge confirms nothing is missing.

03
Code
Fully automated

Build agent implements end-to-end. A user-proxy reads the actual source files: can a user do what was promised? Gaps repaired before exit.

04
Deploy
North-star alignment check

Checks what was built matches what was promised. Then Dockerfiles, docker-compose, and a versioned deploy script for your platform.

What You Get

The process layer Claude is missing.

Claude stays on-process

A SessionStart hook injects an engagement contract every session — so Claude always routes through /spec-gantry, even deep into a long conversation.

One architecture, no drift

All technical decisions live in one file with named section anchors. Every agent reads only the sections it needs. Your data model stays consistent across capabilities.

Fix the right thing

When something breaks, an investigate agent classifies root cause first — code bug, spec gap, or requirement drift. The right phase gets fixed, not just re-run.

Cost per capability

/track-cost shows cost by capability, by phase, by CWJ step — with outlier flags on capabilities that ran repair cycles.

Works on existing codebases

The reverse-engineer agent synthesises a north star, architecture, and capability stubs from your existing source files. Drop it into any project.

Zero dependencies

Runs entirely inside Claude Code. No npm install, no API keys, no services. Two commands and you're running /spec-gantry in any project.

The Dashboard

One command for the entire pipeline.

Run /spec-gantry at the start of every session. It reads your project state, shows where every capability stands, and routes you to the next action.

Type [>] to auto-run — spec and build all remaining capabilities, pausing only at genuine decision points.

  • New project, existing codebase, daily work, bugs — same command
  • Auto-continue runs until a real decision, not a fake pause
  • All progress on disk — resume any session from exactly where you left off
