The Problem

AI coding tools are fast.
That speed is also the risk.

When your AI can write 500 lines in a minute, it's easy to start building before you've answered the hard questions. Three hours later you're refactoring in code review — or worse, starting over.

The team already knows the right process. SpecGantry makes it impossible to skip it.

Speed without direction

AI writes code instantly, but it doesn't know if you're solving the right problem until after the spec review.

Architecture discovered too late

Violations found in code review cost 10× more to fix than violations caught at spec time.

Invisible token spend

AI loads hundreds of lines of context per feature with no visibility into what each story actually cost.

Arch drift across stories

Each story re-describes entities, permissions, and contracts. Three stories later, the data model has three different definitions of the same entity.

The Solution

Architecture once. Stories that reference it. Code that reads only what it needs.

v5 separates architectural knowledge from story navigation. Shared artifacts are written once during ideation and referenced by every downstream agent — no duplication, no drift, no full-context loads. Session-level engagement hooks keep Claude on-process across context resets.

Key Features

Everything you need to ship AI-assisted features with confidence

Engagement Hooks — Claude Stays On-Process

Project setup writes a SessionStart and PostCompact hook into .claude/settings.json. Claude Code injects the engagement contract automatically on every session start and after every /compact — so Claude never forgets to route through /spec-gantry, even deep into a long conversation.

Shared Architecture Layer — Written Once, Referenced Everywhere

v5 introduces a dedicated specs/architecture/ layer: entities, actors, contracts, patterns, UX conventions, and deployment configuration live in six structured files. Every story spec references them via a reads: block — never duplicates them. Architecture drift across stories is structurally impossible.

How the architecture layer works →

Specs Before Code — Hard Gates

No code is written until a complete story spec exists, arch references resolve, and every guardrail is checked. The gate reads the filesystem — not a flag, an artifact.

Slim Story Specs — ≤60 Lines

Each story spec is a navigation map, not a knowledge dump: criteria, interfaces, permissions, state, data — all as references. Max 60 lines enforced. Build agents load only the sections they need.

Self-Healing Gap Flows

When story-spec finds a missing arch section (P0) or development finds a wrong contract (P1), SpecGantry fills the gap automatically and resumes. No manual intervention. No pipeline stall.

intent.md — Why Every File Exists

Every story gets a 2-paragraph intent.md that grounds development judgment calls. Every source file gets an @intent anchor. Investigation never loads full specs to understand purpose.

Reverse-Engineer Existing Codebases

SpecGantry analyzes your existing code and generates the full architecture layer, story list, intent files, stub specs, and anchor tags. Bring structure to legacy projects in one command.

See reverse engineering →

Full Token Cost Visibility

Every agent invocation logs tokens used, model, and cost. Dashboard shows spend by phase, story, and release. Targeted reads cut development-phase token usage by 40–60% vs earlier versions.

See cost tracking →

Quick-Start Mode — Simple Apps in 3 Questions

SpecGantry detects simple projects automatically — no auth, no AI, single actor, three or fewer capabilities. For those, it sets smart defaults for all architecture decisions and asks only three focused questions: tech stack, deployment, and story list. Same complete artifacts as full ideation, a fraction of the time.

Test Plan — Verify What Was Built

The build agent writes a test plan to build-report.yaml — one shell command per observable acceptance criterion. After a build completes, SpecGantry offers to run them against your app. The investigation agent runs the same plan first when you report a bug, pinpointing the failing criterion before reading any source code.

Crash-Safe Resumption

Every write is gated by a state flag. P0/P1/P2 routing detects incomplete state and recovers automatically on next run — no lost work, no manual cleanup, no corrupted pipeline.

2 Skills · 7 Agents

One command. Every workflow covered.

/spec-gantry
Your Single Entry Point — New project, existing codebase, daily work, bug fixes, new features. Run this at the start of every session. SpecGantry reads your state, shows the full pipeline dashboard, and routes you to the right agent automatically.
/track-cost
Cost Breakdown — Real token counts and dollar costs by phase and story. Input, output, cache write, and cache read shown separately. Know exactly what each feature cost before you deploy.

The 7 Agents

Ideation
Sonnet
Two-beat conversational session. Auto-detects simple projects and routes them through quick-start mode (3 questions, smart defaults). For complex projects, matures the idea through 10 topics — vision, users, constraints, tech stack, guardrails, configuration, story list, UX model, and deployment target. Seeds all 6 architecture artifacts and intent.md per story. Partner, not decision-maker.
Story Spec
Sonnet
Finalizes intent.md, validates arch references, writes a slim story-spec.md (≤60 lines) with a reads: block. Signals arch gaps (P0) when a required section is missing. Self-reviews before setting spec_done:true.
Development
Sonnet
Builds one story end-to-end from intent.md + story-spec.md. Reads only the targeted arch sections declared in reads: (~130 lines total). Implements a /health endpoint as the first backend route. Writes @story, @intent, @entry, @contract anchors in every file. Writes a test_plan to build-report.yaml — one shell command per observable criterion. Signals spec gaps (P1) when a contract or entity is wrong.
Deployment
Sonnet
Generates a versioned deploy.sh covering build, version stamping, migrations, and infrastructure deployment for all stories. Supports --dry-run. Merges gap specs before release.
Investigation
Haiku
Read-only. Given a bug report or enhancement request, first checks the app is running (health gate), then runs the test plan to pinpoint the failing criterion before touching source code. Uses @story, @intent, @entry, @contract, @gap anchors for precise navigation. Returns a structured findings report confirmed with you before any changes.
Reverse Engineer
Sonnet
Analyzes an existing codebase and synthesizes the full SpecGantry project structure — architecture layer (all 6 artifacts), story list, intent.md per story, stub story-specs with reads: blocks, build-report stubs, and anchor tags in source files.
Ideation (gap mode)
Sonnet
Invoked automatically when story-spec or development signals a missing arch section. Adds or updates the missing entity, actor, contract, pattern, or UX section — and updates the Artifact Index. Returns control to the pipeline without user intervention.
