The Problem

You write the spec.
Claude ignores half of it.

Claude Code moves fast — but it has no memory of what you decided in ideation, no idea what the spec says about error states, and no way to know if the user experience it just built matches what you promised. You find the gaps at code review. Or in production.

SpecGantry gives Claude a structured process it can't skip — and enforces it at every phase.

Building before speccing

Missing loading states, ambiguous error messages, and dead ends after failure — caught at spec time, not code review.

Architecture drift

Three capabilities in, the data model has three conflicting definitions of the same entity. One architecture file, written once, prevents this.

Invisible AI spend

No visibility into what each capability cost, how many cycles it took, or which one ran three code repair loops.

Fixing the wrong thing

A spec gap sent to the code repair loop will never converge. Classify first — code bug, spec gap, or requirement drift — then fix the right phase.

How It Works

Every phase is reviewed before the next one starts.

Before anything is written or built, an independent agent asks what would block the next phase. A second agent resolves every gap. A third asks: would the next phase still be blocked? Gaps caught at ideation cost nothing to fix. Gaps caught at spec save a full build cycle.

Challenge
Adversarial agent asks what would block the next phase — specific to this project, not a checklist. Never writes files.
▶
Write / Build
Every challenge resolved. Ideation: user answers all questions per round. Spec: agent writes developer contract. Code: plan then build.
▶
Judge
One question: "Would the next phase be blocked?" CLEAR → loop exits. BLOCKED → gaps fed back for another cycle.
✅ CLEAR — phase exits
🔄 BLOCKED — next cycle
⚠ CAPPED — max cycles, user decides

What You Get

Claude that follows your process,
not just your last message

Claude stays on-process

A SessionStart hook injects an engagement contract on every session — so Claude always routes development through /spec-gantry, even deep into a long conversation.

One architecture file, no drift

All technical decisions — data model, actors, API interfaces, deployment — live in one file with named section anchors. Every capability reads only the sections it needs. Three capabilities in, your data model still has one definition of every entity.

How the architecture file works →

Hard gates between phases

No code is written until a spec exists that was machine-challenged and user-approved. The gate reads the filesystem — not a flag.

Fix the right thing

When something is wrong, the investigate agent classifies the root cause first: code bug, spec gap, requirement drift, or new capability. The right phase gets fixed — not always code.

See what each capability cost

Token usage logged per agent run. The insights view shows iteration counts, challenge density, and which capability ran three repair loops and why.

See developer intelligence →

Works on existing codebases

Drop SpecGantry into any project. The reverse-engineer agent synthesises a north star, architecture, and capability stubs from existing source files — with anchor tags.

Two Commands

One command to run your entire SDLC

/spec-gantry
Your single entry point. New project, existing codebase, daily work, bug fixes, new capabilities. Run this at the start of every session. SpecGantry reads your state, shows the pipeline dashboard, and routes you to the right action.
/track-cost
Developer intelligence. Cost by Challenge/Write/Judge, by capability, by release. Iteration counts, challenge density, outlier detection. Know exactly what each capability cost and why.
